    Known Issues/Incidents

    1. Introduction¶

    IOM plays major / imp part of sales application as per business point of view. Siebel/CRM is frontend of IOM uses IOM to communicate multiple backend (surrounding apps). It doesn’t have its own UI but works underneath the Siebel as it is integrated or embedded part of application. Order management is the sales process which helps in generate revenue with the help of each product which organization introduces. It’s the medium to serve every request raised by agents through CRM tool. It integrates with all the surrounding application to map the data to avoid business loss in any aspect such as it verifies any kind of data related to order with different applications which is related to an end customer information and process the new request. It’s the data bus which check in every point of application to verify data and stops / stuck in case any issue occurs while process is running.

    Few services directly process orders to Online and Batch domain without using OAG or bypass OAG. (Waiting for document from Ilham)

    IOM involves series of synced steps and applications or touchpoints which helps subscriber to order or chose correct product. It also helps business to create inventory of end customers (subscriber) and sales record. Few orders are getting completed as the same time of requests and few runs at night to fulfil the requirement.

    Surrounding Applications: Air → Credit (Huwaei) HLR → Network FOSS → Repository RBM → Billing HBASE →Hadoop DSA → SIM allocation

    Below are oracle products been used as solution to Indosat.

    • Oracle SOA Suite for Oracle Middleware (OSB - Oracle Service Bus) 12.2.1 • Oracle API Gateway (OAG) 11.1.2.4.0

    2. IOM Architecture and surrounding¶

    → Siebel is the frontend for IOM and all the requests comes from Siebel application process orders → Order management doesn’t have it’s own UI but it is integrated part of Siebel application a. IOM is middleware level for applications (Oracle fusion) b. We have 2 Siebel CRM, 1 is for retail which is Siebel Icare and 2nd is for business which is Siebel Catalyst c. CRS and IOM both works under Siebel Icare and both are different apps d. Request process flow

    1. Siebel Icare (Retail PP)
    2. IOM middleware (Oracle Fusion)
    3. Network Elements (HLR, AIR)
    4. Boundary System for billing (RBM)
    5. Next-Gen SSP (NGSSP - Product catalogue)

    3. IOM Workflow¶

    1. Request comes from Siebel to IOM OAG → 2 Gateway OAG (called as proxy api oag or gateway api)
    2. Request pass through all domains/services
    3. New IOM has 4 domains

    3.1 Batch domain (Batch order)¶

    Batch domain used for mass orders, comes from siebel. process from system It has 3 nodes → 1 is admin 2 for MS It’s same as like online domain but it runs at night and handle mass transaction and order triggered by RBM Batch orders initiated after request raised in Siebel such as terminate orders but doesn’t work at the same time but it runs at night

    3.2 Online domain (Online order)¶

    All order requests come from Siebel, process from user It also has 3 nodes → 1 is admin 2 for MS Online transaction comes from Siebel directly is current process runs at the same time of request

    3.3 ESB API domain¶

    Pre-Request order will be passed through this domain before going to Batch or Online domain We need this app because Siebel doesn’t validate any order in Online & Batch so ESB API validates all the requests come from Siebel It has 5 servers to serve 1 is admin and 4 MS

    3.4 Standalone domain¶

    It’s used for other services such prepaid orders Siebel has embedded application which has prepaid screen and application running. It has many application base on java such as prepaid app, billing on mail It’s not for orders but it is to see /validate the other entity in data in surrounding apps and responds back to Siebel (frontend) It has 6 servers 1 serves as admin and 5 are MS

    4. Database¶

    1. Online and batch domain has 1 DB
    2. ESB and Standalone has different DB apart from IOM
    3. XPTIOM DB captures all the online & batch domain orders
    4. OPS DB is used for ESB and Standalone

    5. IOM Infrastructure Overview¶

    a. Difference between old IOM and New IOM is having product catalogue from NGSSP (New Generation Supply Side platform) b. Product coherence is added to new IOM c. Old IOM was having catalogue which was presented in xml format and it was available in IOM DB itself which is mapped with different surrounding apps. d. Surrounding in New IOM work sequentially, as per defined in NGSSP catalog. e. New IOM has NGSSP which is providing product catalogue separately which is helping to find particularly surrounding and sending request to them only f. IOM is connecting NGSSP through http based g. When request goes to NGSSP → It gives information based on product as per document, also IOM request NGSSP to provide product definition

    1-Request from Siebel received in IOM as payload , 2-IOM server redirect the request payload to NGSSP for validating the payload , and to provide the surrounding sequence to follow to complete the request. 3-NGSSP Sent the payload again to IOM servers,where service bus having packages respond to the request. 4-IOM send it to the surrounding to validate and update their db. 5-Oonce the surroundings complete the task the payload is send to iom again. 6-IOM send the completed payload to Siebel updating about the process completion.

    Error Encountered- 1-IOM server maybe down which cause the failure of package transmission. 2- payload is not properly created in Siebel causing error in NGSSP validation or error when Validating from Surrounding. 3- DB errors

    Service bus- service bus is the code repository for IOM (i.e middleware)

    It contains the code for the front end as well as code for validation to NGSSP

    We can check service bus using weblogic console Service bus is a concept of fusion middleware We have service bus for online, batch and API domain Standalone domain use java repository to store code

    MW servers are basically supporting service bus If service bus got down- we can’t see data from client side as well can can’t send data to NGSSP Is any repository got down- we got warning alert in weblogic console

    API architecture- API is a request validator, check request before it reach to online, batch or standalone domain We have 5 ms for API domains,and 1 db for configuration storage 10.34.80.181 is load balancer. Weblogic console for monitoring

    6. IOM Standalone¶

    We are not using oracle fusion in standalone but using Java based code. Path for the all services in server is as below with .war extension.

    /u01/app/config/domains/prdstalesb_domain/servers/AdminServer/upload/

    Below are the applications running in Standalone domain

    Prepaid App¶

    a) Main function of standalone domain is Prepaid App. b) Standalone is mostly used to show the screen (This is present info whatever is required for order mgmt). c) Data is getting presented in standalone then API gets the data from surrounding and presented to the standalone. AIR, HLR and RBM. d) If the standalone goes down then prepaid screen will show timeout error. e) How to check prepaid app screen is running or not.

    f) We can login to standalone weblogic admin and check healthcheck status. g) API, Online & Batch domain service bus to put repository record h) Standalone repository record will be available in weblogic so service bus is not available in standalone but it has java based record. i) We have 1 admin and 5 managed service nodes in standalone environment.

    j) State is active and zero session running then MS is not working and we need to check from server.

    k) If the PrepaidApp.war is gone then we’ll not be able to see the prepaid screen but it will be available from another MS.

    CC Portal¶

    CC portal application also running in standalone domain to monitor all the transaction and re-proecess error for our internal support team.

    The Batcher¶

    The Batcher is also app running in standalone too and it doesn’t have any kind of screen/display/UI but it runs as service but it has important role when the order getting created using batch process which is different than online domain.
